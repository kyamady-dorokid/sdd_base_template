# コミット方針（Commit Policy）

## 基本ルール

**ClaudeCode は自動でコミットしない。** フェーズ区切りでコミットを「提案」する。

- 実装者が内容を確認し、コミットの実施は**人間が判断・実行**する。
- 希望があれば「自動コミットモード」への切替も可能（その旨を明示して依頼すること）。

---

## コミット推奨タイミング

| タイミング | 内容例 |
|---|---|
| フェーズA完了 | `docs/architecture/` 初期ドキュメント作成後 |
| フェーズB/C完了 | SDD設定ファイル・overlayルール追加後 |
| 仕様化完了 | `requirements.md` / `design.md` / `tasks.md` 確定後 |
| 実装完了 | 単体テストパス・`test-results.md` 記録後 |
| 結合試験完了 | `integration-test-checklist.md` チェック完了後 |

コミット推奨時は以下を明示する:
1. コミット対象ファイル
2. 推奨コミットメッセージ（例）
3. 「自動コミットも可能です」の案内

---

## コミットメッセージ規約

Conventional Commits に準拠:

```
<type>(<scope>): <概要（日本語可）>

<本文（任意）>
```

| type | 用途 |
|---|---|
| `feat` | 新機能 |
| `fix` | バグ修正 |
| `docs` | ドキュメントのみの変更 |
| `refactor` | リファクタリング |
| `test` | テストの追加・修正 |
| `chore` | ビルド設定・依存関係・CI等 |

scope 例: `core`, `api`, `ui`, `db`, `sdd`, `docs`（プロジェクトに応じて定義）

---

## 自動コミットモードへの切替

タスク依頼時または作業中に以下のように指示することで自動コミットに切り替えられます:

> 「このタスクはコミットまで自動で進めてください」

自動コミット時も:
- git diff と対象ファイルをメッセージで確認提示してから実行
- force push・履歴改ざん（--amend、reset --hard）は行わない
- コミット前に `git status` を確認し、意図しないファイルが含まれていないか確認する

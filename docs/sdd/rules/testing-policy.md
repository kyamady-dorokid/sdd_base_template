# テスト方針（Testing Policy）

## 基本原則

1. **単体テストは必ず実装コードとセットで書く。** テストなし実装は受け付けない。
2. **TDD（テスト駆動開発）で進める。** テスト項目 → テストコード（RED）→ 実装（GREEN）の順。
3. **テスト項目は仕様ドキュメントから導出する。** `requirements.md` の受け入れ基準をテストケースに落とす。
4. **実行したテストの結果は `docs/specs/<task>/test-results.md` に記録する。**
5. **結合試験項目は `docs/specs/<task>/integration-test-checklist.md` に残し、人間が確認する。**

---

## TDD の進め方

```
仕様（requirements.md）
  │
  ▼
テスト項目を文章で列挙（テスト設計）
  → docs/specs/<task>/test-results.md の「テスト項目」欄に先行記載
  │
  ▼
テストコードを実装（RED）
  → テストを実行してすべて失敗することを確認
  │
  ▼
実装コードを書く（GREEN）
  → テストをすべてパスさせる
  │
  ▼
リファクタリング（REFACTOR）
  → テストが引き続きパスすることを確認
  │
  ▼
test-results.md に実行結果を記録
```

---

## テスト項目の設計指針

- **正常系**: 仕様に定義された正しい入力に対して期待通りの出力が返ること
- **境界値**: 入力の上限・下限・空・null などの境界条件
- **異常系**: 不正な入力・存在しないデータ・権限なしアクセス等
- **仕様の受け入れ基準（EARS形式）を1条件1テストで対応させる**

---

## 初回実装時の技術要件確認

新機能を初めて実装するフェーズの場合、実装前に `tech-requirements.md` を作成して確認する。
（テンプレート: `docs/sdd/templates/tech-requirements.md`）

確認事項の例:
- 利用する言語・FW・ライブラリのバージョン制約
- 既存コードとの結合方法（モデル・コンポーネントの再利用可否）
- テストフレームワーク・実行方法
- 環境固有の制約（DB・言語ランタイムのバージョン等）

---

## test-results.md の記録内容

（テンプレート: `docs/sdd/templates/test-results.md`）

- タスクID・日付・実行者
- テスト項目一覧（項目名・期待結果・実際結果・合否）
- テストコマンド・実行環境
- 失敗があった場合は原因と対処を記載

---

## integration-test-checklist.md の記録内容

（テンプレート: `docs/sdd/templates/integration-test-checklist.md`）

- **人間が手動で確認すべき項目を列挙する**
- ブラウザ操作・サービス間連携・DB整合性など自動テストが難しい部分
- 確認済みチェックボックス形式（`- [ ] 確認項目`）
- 確認日・確認者の記入欄
